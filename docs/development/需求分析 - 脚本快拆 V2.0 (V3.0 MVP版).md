# 需求分析 - 脚本快拆 V2.0 (V3.0 MVP版)

> [!tip] 文档使命 (V2.0)
> 本文档是“脚本快拆” **V3.0 MVP（垂直打击版）** 的**唯一、完整的权威需求定义**。
>
> 它在 V1.1 版（现已归档）的基础上进行了根本性升级，旨在将我们的产品从“通用V1.0工具”重塑为“V3.0垂直赛道武器”。本文档是V2.0技术方案 和V2.0原型设计 的“需求源头”。

---

## 1. 项目使命 (Project Mission) V2.0

### 1.1 核心目标 (Objective)
V3.0 MVP的核心目标是：**All-in “AI与科技产品评测”垂直赛道**。我们旨在通过“非对称作战”，交付一个“**科技视频文稿提取器**”，解决主流竞品（Descript, Sonix） 未能解决的P0级痛点，以此验证我们的垂直打击战略。

### 1.2 成功标准 (Success Criteria)
* **定性标准:**
    * 成功上线“形态B：专业模式” 产品。
    * V3.0核心用户（科技创作者） 对“99%+转录准确率” 和“结构化提取” 的价值高度认可（“真摩擦” 被解决）。
* **定量标准 (V3.0):**
    * **P0 价值验证:** “科技评测”模式 的使用率 > “通用叙事”模式。

## 2. 核心用户 (The User) V2.0

### 2.1 用户画像 (Persona) V2.0
* **画像名称:** **“工匠型”科技评测创作者 (The "Craftsman" Tech Creator)**
* **画像简介:**
    * 他们是 V1.0“效率驱动的创作者” 的**专业化子集**。
    * 他们是“AI与科技产品评测”赛道 的“Prosumer”（专业消费者）。
    * 他们对工具的 **“专业性”** 和 **“准确性”** 的容忍度极低。
    * 他们愿意为能解决其“真摩擦”（如手动校对竞品的垃圾文稿、手动提取参数） 的专业工具付费。

### 2.2 待办任务 (Job-to-be-Done) V2.0
> "帮我**极其准确地（99%+）** 转录我的科技评测视频，并**自动提取出**所有‘产品参数’、‘价格’和‘卖点’，交付给我一个可立即用于下一步（如写稿、对比）的‘半成品’。"

## 3. 功能范围 (Functional Scope) V2.0

### 3.1 核心功能 (Core Features) V2.0
* **V1.0 (基础功能):** 提供视频URL/文件上传、结果复制/下载功能。
* **V2.0 (通用赛道):** 保留V1.0的“通用叙事分析”（钩子、核心、CTA） 作为通用选项。
* **V3.0 (垂直赛道 - P0):**
    1.  **“形态B”专业模式选择器:** 在UI上提供“分析模式”选择器，作为V3.0的“UI护城河”。
    2.  **99%+ 垂直转录:** 提供“科技评测” 专属的高精度转录（通过阿里云热词API实现）。
    3.  **V3.0 结构化提取:** 提供“科技评测” 专属的结构化洞察（参数、卖点、价格等）。

### 3.2 用户故事 (User Stories) V2.0

**史诗故事 1: 核心转录与分析流程 (V1.0 - 保持)**
* **用户故事 1.1: 灵活的输入方式** (保持不变)
* **用户故事 1.2: 查看结果** (保持不变)
* **用户故事 1.3: 导出与复制结果** (保持不变)

**史诗故事 2: V3.0 垂直打击工作流 (V2.0 - 新增)**
* **用户故事 2.1: 意图声明 (专业模式选择器)**
    * **作为** 一名“工匠型”科技创作者,
    * **我想要** 在提交视频前，**明确选择**我的分析模式（例如“科技产品评测”）,
    * **以便于** 我能确保系统调用最专业的模型（包括热词库 和专用Prompt）来处理我的内容。

* **用户故事 2.2: 强制选择 (避免P0价值错过)**
    * **作为** 一名首次使用的科技创作者,
    * **我想要** 系统“强制”我（例如通过禁用按钮）在默认的“-- 请选择 --” 之外做出选择,
    * **以便于** 我不会因为“零摩擦”习惯 而错过V3.0的99%+准确率核心价值。

* **用户故事 2.3: 动态结果展示 (不同结构)**
    * **作为** 一名科技创作者,
    * **当我** 选择了“科技评测”模式后，**我期望** 看到的AI分析卡片是“产品参数”、“卖点”和“价格”,
    * **而不是** 通用的“钩子、核心、CTA”。

* **用户故事 2.4: 智能重置 (工作流优化)**
    * **作为** 一名需要批量处理科技视频的创作者,
    * **当我** 点击“再分析一个” 按钮时，**我期望** “分析模式选择器” 自动记住我上一次的选择（“科技评测”）,
    * **以便于** 我在连续分析时，无需重复设置我的专业模式。

### 3.3 排除项 (Out of Scope)
*(V1.0的排除项 保持不变：无用户系统、无在线编辑、无高级导出、无批量处理。)*
* **用户账户与历史记录:** MVP将不包含用户注册、登录以及历史分析结果的保存功能。
- **在线编辑功能:** MVP不支持在应用内对分析结果进行任何形式的编辑。
- **高级导出格式:** MVP将不支持 `.pdf`, `.docx`, `.srt` 等格式的导出。
- **批量处理功能:** MVP将不支持一次性上传或提交多个视频进行处理。
- **复杂分析功能:** MVP将不包含“爆款预测”、“金句提炼”、“情感分析”等高级AI功能。
* **V3.0 P1 排除项:** “内容复用工厂”（如一键生成Shorts脚本） 明确排除在P0范围外。

## 4. 非功能性要求 (Non-Functional Requirements) V2.0

### 4.1 性能 (Performance)
- **响应时间:** 对于一个5分钟以内的标准短视频，从用户提交到返回分析结果的总时长，应控制在 **3分钟** 以内，以保证良好的用户体验。

### 4.2 约束与依赖 (Constraints & Dependencies) V2.0

#### 4.2.1 依赖项一：语音转文字 (ASR) 服务
* **最终选型:** **阿里云通义听悟 (Alibaba Cloud Tongyi Tingwu)**。
* **V3.0 NFR (准确率):** Coprocessor **必须** 在调用“阿里云ASR”时，集成其“自定义热词” 功能。
* **V3.0 NFR (路由):** Coprocessor **必须** 根据 `analysis_mode` 动态注入正确的 `HotwordListID`（例如，为“科技评测” 注入“科技词库ID”）。

#### 4.2.2 依赖项二：大语言模型 (LLM) 服务
* **主选方案 (Primary):** **DeepSeek Chat (`deepseek-chat`)**。
* **备用方案 (Failover):** **Kimi Chat (Moonshot AI)**。
* **V3.0 NFR (可靠性):** V3.0架构（`LLMExecutionService`） **必须** 保留V1.0的“主备切换”（Failover）逻辑。
* **V3.0 NFR (路由):** V3.0架构（`LLMTrackRouter`） **必须** 根据 `analysis_mode` 调用不同的Prompt模板。
* **V3.0 NFR (数据契约):** Coprocessor **必须** 支持“不同赛道，不同（特化）结构” 的JSON输出，并强制使用LLM的JSON模式。
